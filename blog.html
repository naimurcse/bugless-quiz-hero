The difference among map vs forEach vs filter vs find.

map

  • Purpose: "map" is used to transform each element of an array and create a new array with the same number of elements.
  • Behavior: It iterates over every element of the array, applies a callback function to each element, and collects the return values of the callback in a new array.
  • Return: It returns a new array. It doesn't modify the original array.

forEach

  • Purpose: "forEach" is used to iterate over an array and perform an action on each element without creating a new array.
  • Behavior: It iterates over the array, applies a callback function to each element, and does not return a new array. It is mainly used for its side effects, like logging or updating values.
  • Return: It doesn't returns a new array. It doesn't modify the original array.

filter

  • Purpose: "filter" is used to create a new array containing elements that meet a certain condition.
  • Behavior: It iterates over the array, applies a callback function to each element, and includes elements in the new array for which the callback returns true. It doesn't modify the original array.
  • Return: It returns a new array. It doesn't modify the original array.

find

  • Purpose: "find" is used to retrieve the first element in an array that meets a specified condition.
  • Behavior: It iterates over the array, applies a callback function to each element, and returns the first element for which the callback returns true. It stops iterating after finding the first match.
  • Return: It returns the first matching data. It doesn't modify the original array.

The difference among cookies vs local storage vs session storage.

Cookies

  • Purpose: Cookies are small pieces of data that can be stored in a user's browser. They are typically used for various purposes, including session management, user preferences, and tracking.
  • Data Persistence: Cookies can have both a session scope (they are deleted when the browser is closed) and a persistent scope (they are stored on the user's device for a specified duration).
  • Size Limit: Cookies are limited to about 4KB of data per domain.
  • Accessibility: Cookies can be accessed both on the client and server-side. They are automatically sent with HTTP requests to the domain they are associated with.

Local storage

  • Purpose: Local Storage is a client-side web storage mechanism that allows developers to store key-value pairs of data in a user's browser for a longer period, often for application settings, user preferences, and offline caching.
  • Data Persistence: Data stored in Local Storage remains even after the browser is closed and is persistent across sessions. It is not limited to a single session.
  • Size Limit: Local Storage has a larger storage capacity compared to cookies, usually around 5-10MB per domain.
  • Accessibility: Local Storage data can only be accessed on the client side via JavaScript and is not automatically sent with HTTP requests to the server.

Session storage

  • Purpose: Session Storage is similar to Local Storage but has a shorter lifespan. It is often used for temporarily storing data that should be available only during a single browser session.
  • Data Persistence: Data stored in Session Storage is accessible only during the current browser session. When the browser is closed, the data is deleted.
  • Size Limit: Similar to Local Storage, Session Storage typically has a storage capacity of around 5-10MB per domain.
  • Accessibility: Session Storage data is accessible on the client side using JavaScript, and it is not automatically sent with HTTP requests to the server.

What is even loop and how does it work?

Call Stack

  • JavaScript code execution begins in the call stack. The call stack is a data structure that keeps track of the currently executing function or method. When a function is called, it's pushed onto the call stack, and when it returns, it's popped from the stack.

Callback Queue

  • Asynchronous operations, such as user interactions, network requests, and timers, are initiated and their corresponding callbacks are placed in a callback queue. This queue holds tasks that need to be processed once the call stack is empty.

Event Loop

    The event loop continuously checks two things:

  • Is the call stack empty? If it is, the event loop looks at the callback queue.
  • If there are tasks in the callback queue, it takes the first task (callback) and pushes it onto the call stack for execution.

Execution

  • The task in the call stack is executed, and this process continues in a loop. The event loop ensures that asynchronous tasks are executed in the order they were added to the queue, and it prevents blocking the main thread.

Non-Blocking

  • The key to the event loop's non-blocking behavior is that it allows JavaScript to continue executing other code or processing user interactions while waiting for asynchronous tasks to complete. This prevents the application from freezing or becoming unresponsive.
